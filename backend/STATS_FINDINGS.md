# Stats Collector Findings

## 1. Источник net_asset

### Проблема
`get_balance()` (через executor) возвращает **только free** — средства не заблокированные в ордерах. Для сетки с 10 уровнями это может быть 50% от реальных активов.

### Решение
Используем `exchange.fetch_balance()["total"]` — сумма `free + used`. Для paper mode `get_balance()` уже возвращает всё (нет locked).

### Формула
```
net_asset = base_total * ticker.mid + quote_total
```

## 2. Источник profit_math

### Проблема
Два кандидата:
1. `sum(order.profit for order in state.orders)` — кумулятивный per-level, **перезаписывается** каждый цикл в `_calc_profit()`. При удалении уровня (bloat cleanup) теряется.
2. `state.realized_pnl` — инкрементальный счётчик, увеличивается на каждый fill в `on_order_filled()`.

### Решение
Используем `state.realized_pnl`. Он:
- Не зависит от количества ордеров в state
- Не теряется при bloat cleanup
- Совпадает с `grid.realized_pnl` в БД (persisted каждый тик)

## 3. Мульти-сетка на одном аккаунте

### Проблема
Если два грида работают на одном exchange_account, `fetch_balance()` вернёт одинаковый баланс для обоих. `net_asset` будет задвоен.

### Текущее поведение
Сборщик делает **один запрос на аккаунт** (`account_ids_seen` set), но `GridStatSnapshot` записывается для каждого грида с тем же `net_asset`. Это корректно для текущей архитектуры (1 grid = 1 account), но нуждается в доработке если появится multi-grid per account.

### Как это влияет
- `earnings_total_pct` будет занижен (profit делится на полный баланс аккаунта, а не на долю сетки)
- `profit_drift` будет включать изменения от другой сетки
- **Для текущего деплоя это не проблема** — каждый аккаунт привязан к одной сетке

## 4. Первый замер (skip)

Сборщик **пропускает** первый замер для каждого аккаунта:
- `net_asset_sag` требует предыдущее значение
- `profit_drift` требует первое значение

При рестарте воркера `_prev_net_assets` и `_first_net_assets` сбрасываются — первый цикл снова будет пропущен. Это означает потерю ~1 минуты данных при каждом рестарте, что приемлемо.

## 5. profit_drift: формула и смысл

```
profit_drift = profit_math - (net_asset - first_net_asset)
```

- `profit_math` — сколько бот **насчитал** прибыли
- `net_asset - first_net_asset` — сколько **фактически** изменился баланс

Если drift > 0: бот считает прибыль больше, чем реальное изменение баланса. Причины:
- Курс упал (unrealized loss на базовом активе)
- Комиссии не учтены в profit_math

Если drift < 0: баланс вырос больше, чем бот насчитал. Причины:
- Курс вырос
- Внешнее пополнение

**Max drift** — максимальное абсолютное расхождение за весь период. Сигнал проблем если растёт линейно.

## 6. Retention plan

Снимки каждую минуту = ~1440 записей/день на грид. За месяц ~43200 записей.

**Текущий план:** не удалять. Таблица компактная (8 numeric полей + timestamp + FK), индексы по (grid_id, time) обеспечивают быстрые range-запросы.

**Когда потребуется:** если грядов станет >50, рассмотреть:
1. Downsampling: агрегировать до 5-минутных/часовых интервалов для данных старше 7 дней
2. Partitioning: по месяцам
3. TTL: удалять старше 90 дней

## 7. API endpoints

| Endpoint | Описание |
|---|---|
| `GET /api/grids/{id}/stats` | Computed metrics: earnings, drift, drawdown |
| `GET /api/grids/{id}/stats/series?hours=24` | Raw time series for charts |

## 8. Что НЕ покрыто

1. **Retention/cleanup** — нет автоматического удаления старых снимков
2. **Per-grid net_asset isolation** — при multi-grid на одном аккаунте
3. **Commission tracking** — profit_math не учитывает комиссии, drift отражает их косвенно
4. **Alerting** — нет уведомлений при аномальном drift или drawdown
