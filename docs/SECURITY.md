# SECURITY

## Обязательный минимум перед продом

- Сменить все дефолтные секреты в `.env`
- Включить 2FA всем `admin` и `superadmin`
- Не использовать live-ключи Binance без теста на `paper` и `testnet`
- Ограничить CORS только своим доменом
- Поставить HTTPS и reverse proxy

## Работа с API-ключами

- Ключи хранятся в БД только зашифрованными
- Никогда не логируй `api_key` и `api_secret`
- Для каждого аккаунта желательно отдельные ключи с минимальными правами
- При подозрении на компрометацию сразу удалить ключи на Binance и перевыпустить их

## Работа с сетками

- Начинать только с маленького `lot_size`
- Не запускать `live` без наблюдения за первыми циклами сетки
- Вводить лимиты на количество уровней и размер позиции на стороне эксплуатации

## Что мониторить в аудит-логе

- массовые неудачные логины
- внезапное выключение 2FA
- создание новых админов
- изменение ролей
- удаление биржевых аккаунтов и сеток

## Что делать при инциденте

1. Остановить все `live` сетки
2. Отозвать скомпрометированные Binance API-ключи
3. Сменить `JWT_SECRET` и `ENCRYPTION_KEY`
4. Проверить `audit_log`
5. Пересоздать доступы администраторам

## Рекомендации на вторую итерацию

- rate limiting через Redis middleware
- rotation refresh-токенов
- отдельный secrets manager
- уведомления в Telegram о критичных действиях
- более строгие лимиты на RBAC и approval flow для live-запуска
