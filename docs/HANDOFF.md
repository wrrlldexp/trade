## Архитектурные решения, принятые до тебя (НЕ переспрашивай)

Это всё уже обсуждено с пользователем — менять без явной просьбы нельзя:

1. **Стек:** Python 3.12 + FastAPI + ccxt + PostgreSQL 16 + Redis 7 + React 18 + Tailwind. Без Celery — Redis pub/sub достаточно.
2. **Биржа:** Binance (через ccxt). Поддержку других бирж не делаем сейчас, но архитектура через ccxt легко расширяется.
3. **Frontend:** один и тот же код для web и Mini App, отличается только адаптер авторизации. Tailwind с CSS-переменными подхватывает темы Telegram/VK нативно.
4. **Mini App:** обе платформы (Telegram + VK), но без реальной интеграции с initData в этом цикле — фронт **готов** к ней, а подключение делает пользователь, когда зарегистрирует приложения.
5. **Режимы:** paper и live параллельно, переключатель на каждой сетке. Один и тот же код стратегии работает в обоих режимах через интерфейс `Executor`. **Это критично для дизайна Шага 3.**
6. **Несколько биржевых аккаунтов на админа** — в схеме БД через `ExchangeAccount.owner_id`.
7. **Команда:** 2–5 человек. Роли: `superadmin` / `admin` / `viewer`.
8. **Регистрация админов:** только по приглашению. Первый суперадмин — через CLI (`make superadmin`). Открытой регистрации НЕТ.
9. **2FA:** обязательно для ролей admin и superadmin. TOTP через pyotp.
10. **API-ключи биржи:** хранятся зашифрованными в БД (Fernet). Мастер-ключ — в `ENCRYPTION_KEY`. Этот модуль (`app/core/crypto.py`) ещё **не создан** — будет в Шаге 4.
11. **Аудит:** все действия пишутся в `audit_log` (кто, что, когда, с какого IP).
12. **JWT:** 15 минут access + 7 дней refresh.
13. **БД:** PostgreSQL с транзакциями (в старом проекте было MyISAM без транзакций — это был один из главных архитектурных дефектов).
14. **UUID, не int** для всех публичных ID (чтобы по URL нельзя было перебирать).
15. **Воркер:** один процесс, сетки внутри — asyncio таски. Не отдельный процесс на сетку. Управление через Redis pub/sub команды (start/stop сетки).
16. **Хостинг:** локально для разработки, прод решается потом.
17. **Старая БД:** не мигрируем, начинаем с чистого листа. Старый код — это спецификация, не источник данных.
18. **Telegram бот / VK app:** регистрацию делает пользователь сам, я не делаю.

## Что нужно сделать на Шаге 3

**Цель Шага 3:** реализовать ядро торговой стратегии — `GridEngine` — с полным набором тестов. **Без интеграции с биржей**: только чистая логика, которая работает с интерфейсом `Executor`. Реальный Binance подключается в Шаге 4.

### Что должно быть в Шаге 3

1. **`app/strategy/executor.py`** — абстрактный интерфейс `Executor` с методами:
   - `place_order(side, price, amount) -> OrderResult`
   - `cancel_order(exchange_order_id) -> bool`
   - `get_ticker() -> Ticker` (current bid/ask)
   - `get_balance() -> Balance`

2. **`app/strategy/paper_executor.py`** — `PaperExecutor`, который реализует интерфейс через симуляцию: ведёт виртуальный баланс, при движении курса проверяет, исполнились ли лимитные ордера. Это должно работать **полностью без сети**.

3. **`app/strategy/engine.py`** — `GridEngine`, основной класс стратегии:
   - `build_grid(current_price)` — создаёт начальную сетку из ордеров (логика из PDF: ниже курса — buy, выше — sell, с заданным шагом)
   - `on_order_filled(order)` — когда ордер исполняется, ставит зеркальный на противоположной стороне (buy → sell на +profit_step, и наоборот). Это сердце grid trading.
   - `check_boundary(current_price)` — проверка, не вышел ли курс за крайние ордера сетки. Если да — взводит таймер `last_boundary_hit_at`.
   - `should_rebuild(now)` — если таймер взведён и прошло `rebuild_timeout_sec` (60 минут по умолчанию из PDF), возвращает True.
   - `rebuild_grid(current_price)` — отменяет старые ордера, строит новую сетку вокруг нового курса.

4. **`app/strategy/types.py`** — dataclasses/Pydantic модели: `OrderResult`, `Ticker`, `Balance`, `GridParams`, `GridState`. Это контракты между Engine и Executor.

5. **`tests/strategy/`** — **полный набор unit-тестов** на `GridEngine` с `PaperExecutor`:
   - `test_build_grid.py` — корректность построения сетки (правильные цены, правильное количество ордеров сверху и снизу)
   - `test_on_filled.py` — корректность переворота ордеров (buy → sell на +profit, sell → buy на -profit)
   - `test_boundary.py` — детект выхода за границы, взведение таймера, сброс при возврате курса
   - `test_rebuild.py` — корректность перестройки сетки
   - `test_full_scenario.py` — end-to-end сценарий: построили сетку → курс ходит → ордера срабатывают → курс уходит → сетка перестраивается. Проверить итоговый PnL.

### Принципы для Шага 3

- **Никаких сетевых вызовов.** Стратегия не должна знать про Binance, ccxt, http. Только интерфейс `Executor`.
- **Никаких обращений к БД из engine.** Engine принимает state на вход и возвращает state на выход. Сохранение в БД — отдельный слой (это уже Шаг 4).
- **Decimal, не float**, для всех цен и количеств. У питона `float` — плавающая запятая, в финансах это запрещено.
- **Тесты на каждое поведение из PDF.** PDF лежит в… У пользователя в чате был файл `адаптивная_стратегия_основная.pdf` — описание стратегии. Если у тебя его нет, попроси прислать или прочти краткий пересказ ниже.
- **Покрытие тестами не менее 90%** по `app/strategy/`. Запускай `pytest --cov=app/strategy` и убедись.

### Краткий пересказ стратегии из PDF (на случай если файла нет)

1. Робот берёт текущий курс выбранной криптовалюты (BTC).
2. На входе: размер лота, размер прибыли (profit), шаг сетки (step). Шаг и прибыль могут быть **разными** (асимметричная сетка).
3. Робот строит сетку ордеров **симметрично** относительно курса: ниже курса — на покупку, выше — на продажу. Например, при курсе 1000$, шаге 300$ и прибыли 300$:
   ```
   SELL: 1300, 1600, 1900, 2200
   BUY:  700,  400,  100,  (и т.д.)
   ```
4. Когда ордер исполняется, робот ставит зеркальный на противоположной стороне с учётом profit_step. Был buy@700 → ставим sell@(700+300)=1000.
5. Если курс уходит за крайний ордер сетки и **не возвращается в течение 60 минут** — сетка перестраивается вокруг нового курса.
6. Целевая скорость — не менее 600 сделок в час (это требование к производительности — учитывай в дизайне, но в тестах не критично).

## Технические нюансы, на которых легко споткнуться

1. **SQLAlchemy 2.0 async** — в env.py Alembic используется `async_engine_from_config`. Если возникнут проблемы с autogenerate, проверь что все модели импортированы в `migrations/env.py` через `from app.models import *`.

2. **DATABASE_URL** должен быть `postgresql+asyncpg://...`, не `postgresql://...`. Уже так в .env.example, но проверь.

3. **Pydantic v2.** Не v1. Синтаксис другой. `model_config = SettingsConfigDict(...)` вместо старого `class Config`.

4. **Bcrypt в passlib** — на новых версиях passlib бывает warning про `__about__`. Не страшно, можно подавить или обновить bcrypt отдельно.

5. **Volume для node_modules** в docker-compose — `- /app/node_modules` без mount, чтобы локальные node_modules не перетёрли контейнерные.

6. **CORS_ORIGINS** в .env — список через запятую, парсится в `cors_origins_list` в config.py.

7. **На фронте VITE_API_URL=http://localhost:8000** — это URL для **браузера**, не для контейнера. Не путай с DATABASE_URL, который из контейнера в контейнер.

8. **structlog** — настройка в `app/core/logging.py`. Вызывается в начале main.py через `configure_logging()`. Если воркер не логирует — проверь, что в его main.py тоже вызвано.

9. **Decimal vs Numeric.** В моделях `Numeric(20, 8)`. На уровне Python используй `decimal.Decimal`, не `float`. SQLAlchemy конвертирует автоматически, но если float попадёт в Decimal-поле — будут странные баги округления.
